# Plugin Type Re-export Pattern

When a pnpm workspace plugin needs to expose API classes (e.g. `VaultProjectApi`) as importable types from the package root, follow this pattern.

## Problem

Consumers import types from deep paths like:
```ts
import type { VaultProjectApi } from '@cpn-console/vault-plugin/types/vault-project-api.js'
```

This is fragile — it depends on internal file layout and requires subpath `exports` in `package.json`.

## Solution

### 1. Re-export from source `src/index.ts`

```ts
// plugins/vault/src/index.ts
import { VaultProjectApi } from './vault-project-api.js'
import { VaultZoneApi } from './vault-zone-api.js'
export { VaultProjectApi, VaultZoneApi }
```

This causes `tsc` to emit `export { VaultProjectApi, VaultZoneApi }` in the generated `types/index.d.ts`.

**Do NOT edit `types/index.d.ts` directly** — it is regenerated on every build and your edits will be overwritten. The `types/*.d.ts` files are build artifacts, not source files.

### 2. Simplify `package.json` exports

```json
{
  "exports": {
    ".": {
      "types": "./types/index.d.ts",
      "import": "./dist/index.js",
      "default": "./dist/index.js"
    }
  },
  "types": "./types/index.d.ts"
}
```

No subpath exports needed.

### 3. Update consumers to root imports

```ts
// Before:
import type { VaultProjectApi } from '@cpn-console/vault-plugin/types/vault-project-api.js'
// After:
import type { VaultProjectApi } from '@cpn-console/vault-plugin'
```

### 4. Update triple-slash references

```ts
// Before:
/// <reference types="@cpn-console/vault-plugin/types/index.d.ts" />
// After:
/// <reference types="@cpn-console/vault-plugin" />
```

## Verification

After making changes:
1. Run `tsc` in the plugin to regenerate `types/index.d.ts` — verify the `export` statements appear
2. Run `tsc` in consumer packages — verify no `TS2459` or `TS2307` errors
3. Run the full monorepo build: `pnpm -r run build`

## Common Pitfall: editing generated `.d.ts` in Docker CI

When building in Docker, the workspace packages are copied as isolated files. The `types/*.d.ts` files are regenerated by `tsc` during the build. If you edited `types/index.d.ts` directly (instead of `src/index.ts`), the Docker build will overwrite your changes and the build will fail with `TS2459: Module 'X' declares 'Y' locally, but it is not exported.`

**Always put re-exports in the source `.ts` files, never in generated `.d.ts` files.**
